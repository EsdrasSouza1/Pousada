Dores identificadas na entrevista com a gestora

  • Conflitos de agenda e risco constante de reserva dupla (overbooking)
  • Impossibilidade de verificar disponibilidade em tempo real
  • Perda de hóspedes que preferem agendar por canais digitais
  • Nenhum histórico centralizado de reservas e hóspedes
  • Sobrecarga operacional respondendo as mesmas perguntas no WhatsApp

Por que não usar um PMS comercial?

Property Management Systems (PMS) comerciais integram reservas, check-in e faturamento — mas os custos de licenciamento e implantação são incompatíveis com a realidade de uma microempresa familiar.

A solução: um sistema web sob medida, de custo zero de licenciamento, fácil de hospedar e na escala exata do negócio.

Por que PHP puro?

  • Suportado nativamente em praticamente toda hospedagem compartilhada
  • Custo zero de licenciamento e documentação extensa
  • Código legível e mantível por desenvolvedores de nível básico

Por que JSON como banco?

  • Banco inteiro em um arquivo: copiar a pasta = migrar o sistema
  • Zero configuração — nenhum servidor MySQL para instalar
  • Volume da pousada (dezenas de reservas/mês) é plenamente atendido

Núcleo: verificação de disponibilidade
function checkDisponibilidade($quarto_id, $entrada, $saida) {
    $db = lerBanco();
    foreach ($db['reservas'] as $r) {
        if ($r['quarto_id'] != $quarto_id) continue;
        if ($r['status'] === 'cancelada') continue;

        // sobreposição de períodos
        if ($entrada < $r['data_saida'] &&
            $saida   > $r['data_entrada'])
            return true;  // CONFLITO
    }
    return false;  // disponível
}

Versionamento do banco

O campo _version permite evoluir a estrutura dos quartos automaticamente preservando todas as reservas existentes — sem migração manual.

Sistema de Reservas com Calendário Inteligente

O hóspede não consegue nem selecionar uma data ocupada

Pousada Barão
Hóspede escolhe o quarto
Site consulta api_disponibilidade.php via AJAX
Flatpickr pinta de vermelho as datas ocupadas
Hóspede só seleciona períodos livres
Reserva criada como "pendente"

Validação em duas camadas

  • Cliente (JS): bloqueio visual de datas, checkout limitado pela próxima reserva (maxDate), resumo noites × preço em tempo real
  • Servidor (PHP): campos obrigatórios, data no passado, checkout > checkin, capacidade máxima e re-checagem de disponibilidade
  • Mesmo que o JavaScript seja burlado, o servidor nunca aceita uma reserva conflitante

O desafio técnico do calendário

Problema: o <input type="date"> nativo do HTML não permite desabilitar datas individuais.

Solução: biblioteca Flatpickr recebe os períodos ocupados da API e os desabilita como intervalos {from, to} — riscados em vermelho no calendário. Ao escolher o check-in, o sistema ainda calcula o limite máximo do check-out para não atropelar a próxima reserva.

Limitações Reconhecidas e Trabalhos Futuros

Transparência técnica e caminho de evolução

Pousada Barão
Limitações atuais

Senha hardcoded

Senha do admin definida no código-fonte — deve migrar para variável de ambiente com password_hash().

Concorrência no JSON

Sem file locking: escritas simultâneas podem corromper dados. Risco negligenciável na escala atual.

Sem notificação automática

O admin não recebe e-mail/SMS ao chegar nova reserva — o envio do WhatsApp é manual.

Próximos passos

Migração para MySQL

A camada db.php isola o acesso a dados — trocar JSON por PDO/MySQL não altera nenhuma página do site.

Pagamento online

Integração com Pix / Mercado Pago para cobrança no ato da solicitação.

Gestão visual

Calendário de ocupação, gestão de quartos pelo painel e relatórios mensais de receita em PDF.
